    It has never been easier for people to teach themself programming. Whether you are learning from a popular online tutorial, or the CS textbook of your friend. At some point, learning object-oriented programming will almost be an inevidable part of your journey. It was when you first learned that the three basic fundamentals of OOP: "Encapsulation, Inheritance, and Polymorphism". The three great, profound ideas that makes everyone's code better. But the words themself doesn't make much sense to you, so you were given an example:

First let's us take a look at Inheritance. Use the school as an example, everyone is a person, so they would all inherit from the class Person. Such as student and faculty. Student would have special attributes such as GPA, and faculty will have special attributes such as wages. Professor would also inherit the class faculty with additional attributes such as classesTaught, since professor is-a faculty.

    Such simple concept, yet such clever idea. When writing a program, simply describe the logical relationship between each object and you will increase your productivity, increase code reusability, readability, and write less bug-prone code.

    "But what about TA?" some might think. Isn't TA both a faculty and a student? So I would just have the class TA inherit from both Student and Faculty. Right?

    Unfortunatly you can't. But don't worry! As a beginner coder you yet have much to learn. People have invented design patterns that solves many issues like this. As you learn more about OOP, you will be capable of solving these problems, and grow as a programmer.................. Right?

    This is how the world seduce another individual into the pit hole of OOP. When people need to use special design pattern in order to intergrate the idea of TA into the "university system", how much potential do you still see in such design principle? Indeed, OOP program is "easy to implement" as it transfer our interpertation of relationship between real world objects directly into the design of program structure. This has guarenteed that any individual with any mental capability to be able to write OOP program. With a fake promise that OOP would work automatically.

    Some might start arguing, "as long as you follow strictly to the OOP design patterns, you would be able to write OOP code with intuitive structure and high reusability." So here is another example where OOP unexpected bug (Effective Java #16): 

public class MyHashSet<E> extends HashSet<E> {
    private int addCount = 0;
    public int getAddCount() {
        return addCount;
    }
    
    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }
    
    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c);
    }
}

But the code won't work. The error occurs because under super.addAll it adds elements through issuing function call towards add. Based on polymorphism function call towards add will then get transfered into the add function under MyHashSet, therefore allAll will add element twice. This error seemed somewhat obvious, but when the project gets large one can never avoid random problem like that unless he knew every single implementation detail of the project. But wait, isn't one of the clear advantage that OOP ought to bring us is portability. While in reality as the a project gets larger and larger, people will be forbidden from making changes towards parent class since many unexpected error might occur. This type of code base is also known as the "shit mountain" - all those shit will fall on your head when you try to change its center.
